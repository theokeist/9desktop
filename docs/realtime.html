Realtime & animation

“Realtime” in 9DE means: predictable responsiveness under load, not a 60fps demo. Animations are allowed, but they are budgeted and cancelable.

Event sources

  • Input: mouse/keyboard events via libevent (einit, event).
  • Timers: a small scheduler that triggers at a fixed tick (e.g. 16–50ms depending on needs).
  • Background work: workers send messages back to the UI thread (planned: libthread channels or a pipe).
  • Services: state changes stream in from /srv/9de (events file) (planned).

UI thread rule

Do not block the UI thread. Anything that might take longer than a few milliseconds goes to a worker, and the UI gets a state update (progress/result) via a message.

Animation budget

  • Default tick: 20–30 fps for spinners/toasts is enough.
  • Only animate when visible.
  • Prefer “stepped” animation (frame index) over expensive geometry.
  • Coalesce redraws: if 5 things invalidate during a tick, redraw once.

Loading patterns

Use the right pattern for the cost:

  • Spinner: unknown duration, short operations.
  • Progress bar: known work units, longer operations.
  • Skeleton rows: content layout is known, data is loading (best perceived performance).
  • Inline “busy” badge: lists of services/jobs; avoids full-screen blocking.

Example: spinner component

// pseudo: a spinner is just a stateful widget with a phase
typedef struct UiSpinner UiSpinner;
struct UiSpinner{
  int phase;    // 0..11
  int running;  // 1/0
};

void spinnertick(UiSpinner *s){
  if(!s->running) return;
  s->phase = (s->phase + 1) % 12;
  uiInvalidate(s);   // mark dirty; next tick redraws
}

Example: skeleton rows

// draw “Surface” + a few muted bars instead of text
// cheap rectangles; no blur, no gradients
uiDrawSurface(r);
uiDrawBar(r, 12, 10, 240, 10); // title bar
uiDrawBar(r, 12, 28, 320, 10); // subtitle bar
