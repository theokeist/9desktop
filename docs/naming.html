Naming & entrypoints

Goal: one obvious way to start the desktop, one obvious way to control it.

The front door

9de
The user-facing command. Think “startx” + “sessionctl” merged, but Plan 9 style.
9de start
Starts the session by sourcing $home/lib/9de/autostart.rc.
9de devstart
Developer start from repo root (./bin/9de-start.rc).

Components (daemons/apps)

9de-shell
The orchestrator (panel/dash/desktop) and the place that posts /srv/9de.
9de-panel
Panel surface (pinned).
9de-dash
Majestic menu / launcher surface.
9de-desktop
Finder-like desktop surface (planned).
9de-settings
System Settings app (planned).

Session files

$home/lib/9de/defaults.rc
Shipped baseline settings.
$home/lib/9de/config.rc
User overrides.
$home/lib/9de/autostart.rc
Generated startup script (execs 9de-shell).

Control plane

/srv/9de
9p service posted by 9de-shell.
9de ctl ...
Writes to /mnt/9de/ctl (auto-mounts).
9de status
Reads /mnt/9de/status.
9de events
Follows /mnt/9de/events.
Principle: one stable entrypoint (9de), everything else is a component. This keeps docs, scripts, and your mental model clean as the desktop grows.
